Pro_Pack : Animation¶

ProPack animation module deals with the more complex animation functions such as the r9Anim format handling and animation re-director systems.

>>> # import statement for the module via the r9pro decompiler
>>> from Red9.pro_pack import r9pro
>>> r9pro.r9import('r9panim')
>>> import r9panim

This module is extensions to the main animation support through the Red9 StudioPack and requires the StudioPack codebase to run

class AnimMap(filterSettings=None, timerange=(), *args, **kws)¶

Bases: Red9.core.Red9_PoseSaver.DataMap

New base class for the Red9 Animation Format - r9Anim This inherits most of it’s node handling directly from the DataMap class in Red9.core.Red9_PoseSaver making it incredibly flexible when it comes to mapping animation data between nodes and rigs.

Like the PoseSaver this is capable of loading animation data relatively against a given node, allowing you to load libraries of animations one after another in different spaces to that which they were saved.

Vimeo example : https://vimeo.com/167872211

>>> from Red9.pro_pack import r9pro
>>> r9pro.r9import('r9panim')
>>> import r9panim
>>>
>>> dm=r9PAnim.AnimMap()
>>>
>>> # like the pose saver you can load in a given filterSettings object directly
>>> dm.settings.read(os.path.join(r9Setup.red9Presets(),'Red9_DevRig.cfg'))
>>> dm.settings.incRoots=True
>>>
>>> # OR modify it on the fly in the class
>>> dm.settings.metaData=True
>>> dm.settings.nodeTypes=['nurbsCurve']
>>>
>>> rootNode='Red9:Main_Ctr'
>>> filepath='d:/test'
>>>
>>> # saving
>>> dm.saveAnim(rootNode, filepath)
>>>
>>> # basic loading
>>> dm.offset=20
>>> dm.loadAnim(rootNode, filepath)
>>>
>>> # load relative to a given startFrame
>>> dm.loadAnim(rootNode, filepath, loadAsStored=False, loadFromFrm=20)
>>>
>>> # loading relative to a given node
>>> dm.loadAnim(rootNode, filepath, loadAsStored=False, loadFromFrm=70, referenceNode=cmds.ls(sl=True)[0])

Note

like many of these more complex calls this is bound directly to the mRig all the same flags are supported and the behaviour is also extended by way of a postload handler, currently designed to restore exportLoops and audio data that are gathered in the format on save from mRig

>>> mrig=r9Meta.getMetaRigs()[0]
>>> mrig.loadAnimation(filepath)

Note

Internal File Format:

self.poseDict[key][keydata]:

  • file_version : 1.0 = (time, value, inTanType, outTanType, inTanAngle, outTanAngle),
  • file_version : 2.0 = (time, value, inTanType, outTanType, inTanAngle, outTanAngle, inWeight, outWeight, lock, weightLock),

self.poseDict[key][keydata_bases]:

  • [weightedTangents, preInfinity, postInfinity, denseBaked, inttan_accumulated, outttan_accumulated]

Note

because this inherits from the DataMap module certain key internal vars are key to the way the data is loaded.

  • self.matchMethod
    : denotes how the nodes are matched to the internal data, either by base, stripPrefix, index, mirrorIndex, metaData

    see r9Core.matchNodeLists for ore info

  • self.prioritySnapOnly
    : bool - used when loading relative to a given node, limit the nodes which are adjusted in world space

    to a given subset denoted in the filterSwettings object

We don’t pass any data in terms of nodes into the __init__, We’ll deal with those in the saveAnim and loadAnim calls. Leaves this open for expansion

Parameters:
  • filterSettings – specify a filterSettings object, or just use the settings that are bound to the class instance
  • timerange – specify a timerange to process, else use the default Red9 behaviour r9Anim.timeLineRangeGet()
  • restoreTangents – this is an additional **kws and switches OFF the loading of keyframe tangents to speed the load on dense data IF this is set to ‘auto’ (default) then we first test if the keydata_bases was marked as ‘denseData’ and if so we turn OFF the Tangent load for any channels marked as such. The idea being that dense data doesn’t need tangents as it’s baled at key per frame.

loadAnim(*args, **kws)¶

Load a given r9Anim file, this is the same system as the r9Pose file system, all the base handling comes from the r9Pose.DataMap base class

Parameters:
  • nodes – if given load the data to only these. If given and filter=True this is the rootNode for the filter.
  • filepath – file to load - if not given the pose is loaded from a cached instance on this class.
  • useFilter – If the pose has an active Filter_Settings block and this is True then use the filter on the destination hierarchy.
  • loadAsStored – load the anim data from the original time
  • loadFromFrm – if not loadAsStored, load the animation data from a given frame by comparing the internal startFrm stored in the animMap with the given frame
  • loadFromTimecode – load against a given SMPTE timecode / frm binding, calculating the offset of the data to load against a given timecode reference. IF timecodeBinding isn’t set then we gather the reference timecode from the mRig’s internal data, else we use the timecode binding supplied
  • timecodeBinding – (frm, str(‘00:00:00:00’)) Tuple where the first arg is the frame at which the second arg’s SMPTE timecode has been set as reference, basically we’re saying that the timecode at frm is x
  • referenceNode – reference node used as the offset Node for relative loading
  • relativeRots – ‘projected’ or ‘absolute’ - how to calculate the offset, default=’projected’
  • relativeTrans – ‘projected’ or ‘absolute’ - how to calculate the offset, default=’projected’
  • manageRanges – do we (0, 1, 2) = leave, extend or set the timeranges according to the anim data loaded
  • manageFileName – if True and the current Maya scene has no filename other than a blank scene (ie freshly loaded rig) then we take the r9Anim’s filename and rename the Maya scene accordingly
  • applyPose – True by default - apply the static pose stored in the file before loading the animation data
  • keyStatics – if True then we key everything in the data at startFrame so that all non-keyed and static attrs that are stored internally as a pose are keyed.
  • blendRange – None or int : 1 is default. If an int is passed then we use this as the hold range for the data, setting a key at time=startFarme-blendRange to hold the current data before we load the new keys. Note that this also turns on the keyStatics to ensure the data is preserved
  • merge – if True we allow the data to be merged over any current keys, else we cut all keys in the load range first
  • smartbake – only valid if we’re loading with a referenceNode, this tries to respect current keys when doing the processing rather than frame baking

class AnimPointCloud(nodes, filterSettings=None, meshes=None, time=(), prioritySnapOnly=True, *args, **kws)¶

Bases: Red9.core.Red9_PoseSaver.PosePointCloud

This relies heavily on the PosePointCloud base class. This means we can hook directly into the renderMeshes variable to find the Geo for reference

Utilises the AnimPointCloud to allow animation to be re-directed via guide and reference meshes. This generates a spline based guide that you can edit to redirect animation around a scene and because it uses the pointCloud setups nothing is bound to the animation data itself

>>> from Red9.pro_pack import r9pro
>>> r9pro.r9import('r9panim')
>>> import r9panim
>>>
>>> ppt=r9proAnim.AnimPointCloud(nodes='MRig:Locomotion_Root_Ctrl')
>>> ppt.settings.read('Red9_DevRig.cfg')
>>> ppt.time=r9Anim.timeLineRangeGet()
>>>
>>> # build the cloud reference, select the point to use as root reference
>>> ppt.build()
>>>
>>> # apply any offsets you may have made
>>> ppt.apply()
Parameters:
  • nodes – nodes or the root node of the hierarchy to process
  • filterSettings – specific filterSettings object to pass in, else we bind one to settings internally
  • meshes – meshes used as reference Geo when constructing the pointCloud nodes
  • time – timerange to process, if not given we run the r9Anim.timeLineRangeGet() to pull it from the scene
  • prioritySnapOnly – only make ppc points for the filterPriority nodes, this is default for speed
  • self.curveSamples = 10 : the number of cv’s samples on the ground curve
  • self.meshReference = True : add the visual mesh reference
  • self.attachMotionPath = True : build the motionPath editable setup or not
  • self.buildHandles=False : whether to build the additional cv handle nodes
  • self.projectedRootTrans=True : project the PPT root node pivot to world plane translates
  • self.projectedRootRots=True : project the PPT root node pivot to world plane rotates
  • self.pathProjectY=True : project the MotionPath to the world plane at Y=0
  • self.pathAllowHeightVariant=True : allow the path to track height variants
  • self.motionFollowScale=10 : simple scale of the follow object on the path
  • self.fractionMode=True : whether to use the fractionMode on the motionPath built
  • self.step=1 : frame step between compensation keys
  • self.smartbake=False : sparse keyed based on the keys currently on nodes being cached
  • self.smartBakeRef=[] : optional separate nodes to use for the smartBake key extraction process

fbx_to_binder(fbxfiles, binderfile=None, usecurrentscene=False, bake=False, runFilter=True, useTakeData=False, fbx_mode='update_anim', mayafile=True, r9anim=False)¶

Main FBX > Binder call, (PRO_TO_BND) loads the FBX file in, loads the binder in, and transfers the data

..note::
The actual maya file and r9Anim file generation flags are generally NOT used during the Pro_To_BND call, instead during that process we pass the output flags to the ExportManager class to deal with. This is because the exporter calls also manage source-control and all clkeint side specific file management. The file handling in this func is base level only, generating the files as per the fbx filepath.
Parameters:
  • fbxfiles – list of fbx files to transfer. Note that we generally run this on singular file basis and let the Pro_to_BND deal with the file outputs at the client level, allows us to modify the export management on a case by case basis and overload as required
  • binderfile – BND binder file to use in the transfer
  • usecurrentscene – if true use the current scene, which then expects a binder file to be already loaded
  • bake – do we bake the results to the Red9 puppet and clean-up after our-selves? if r9anim is True then this is forced on
  • runFilter – do we run the curve filter on the baked animation to try and optimise the results, removing static data etc
  • useTakeData – if True use the internal fbx take[1]’s timeSpan when setting the ranges
  • fbx_mode – fbx management mode used to get the fbx data over to the bnd skeleton ‘update_anim’ : uses the FBX file import’s UpdateAnimation Only flags - issue here is that if the fbx file has namespaces then the update fails ‘reference_copy’ : uses a reference technique, we reference the fbx file in, then copy the anim data directly using the r9Animfuncs to by-pass FBX import limitations ‘connect_hik’ : references the fbx in then inspects the loaded data for a hik character, if found the code plugs this into the source of our BND hik node
  • mayafile – save the resulting Maya files out. This is a flag so that if this func is being called as part of another process we can choose if we want the Maya file or Not, in the ProToBnd the maya file is managed by the exporter wrap directly and this is not called
  • r9anim – export r9Anim filed out. In the ProToBnd the r9Anim file is managed by the exporter wrap directly and this is not called

Main Maya Animation File > Binder call, (PRO_TO_BND) for animation data on the Red9 Puppet. This requires a properly published Red9 BiPedal Puppet rig and an associated BND binder file

this loads the binder file, references the Maya anim file, and transfers the data either via HIK plugs (default) or by copyAnim calls. This also transfers all exportLoops across and all AudioGrp data

Note

The actual maya file and r9Anim file generation flags are generally NOT used during the Pro_To_BND call, instead during that process we pass the output flags to the ExportManager class to deal with. This is because the exporter calls also manage source-control and all client side specific file management. The file handling in this func is base level only, generating the files as per the input filepath.

Parameters:
  • mayafiles – [] list of animation files to transfer. Note that we generally run this on singular file basis and let the Pro_to_BND deal with the file outputs at the client level, allows us to modify the export management on a case by case basis and overload as required
  • binderfile – correctly setup Red9_Puppet_BND binder file to use in the transfer
  • usecurrentscene – if true use the currently loaded scene, which then expects a binder file to be already loaded
  • output_dir – if given we re-direct the outputs to a new folder, remember we’re loading maya files so we need to do something to stop the process over-writing the input files. if not given we append the scene name with “_converted”
  • bake – do we bake the results to the Red9 Puppet and clean-up after our-selves? if r9anim is True then this is forced on
  • runFilter – do we run the curve filter on the baked animation to try and optimise the results, removing static data etc
  • xfer_mode – transfer mode used to push the animation data over to the bnd skeleton from the incoming PuppetRig animation file ‘reference_copy’ : uses a reference technique, we reference the anim file in, then copy the anim data directly using the r9Animfuncs ‘connect_hik’ : references the anim in then inspects the loaded data for a hik character, if found the code plugs this into the source of our BND hik node
  • run_func – callable, if given we run this function after the bind is done and before the bake is hit, allowing for specific modifications if required, note that the func is passed the instance of the Transfer class object run_func(self)
  • mayafile – save the resulting Maya files out. This is a flag so that if this func is being called as part of another process we can choose if we want the Maya file or Not, in the ProToBnd the maya file is managed by the exporter wrap directly and this is not called
  • r9anim – export r9Anim filed out. In the ProToBnd the r9Anim file is managed by the exporter wrap directly and this is not called

  • removeMeshes – simply delete all meshes as generally they’re not important in anim transfer and slow the file load hugely
  • removeFacial – another huge hit on the BND file load so delete it all if found
